# Правила работы с Pascal/Delphi-проектом

Ты работаешь с Pascal/Delphi-кодовой базой и Markdown wiki проекта.

## Корни проекта
- Корень проекта: `D:/Works/учёба/вкр/finalversion`
- Исходный код Pascal/Delphi: `D:/Works/учёба/вкр/finalversion/data/input_project/src`
- Wiki: `D:/Works/учёба/вкр/finalversion/data/wiki_out/current`

Все чтение файлов и поиск выполняй только внутри этих путей.

## Источники

- `markdown_rag_wrapper` — основной источник для поиска unit/class/method, назначения сущностей и связей.
- `pasls` — уточнение по Pascal/Delphi-коду через LSP.
- Встроенные инструменты OpenCode — чтение и изменение файлов только внутри корней проекта.

## Порядок работы

1. Если вопрос касается архитектуры, назначения сущности, unit/class/method или связей в коде, сначала используй `markdown_rag_wrapper.markdown_rag_search`.

2. Перед первым `markdown_rag_search` в новой сессии один раз вызови `markdown_rag_wrapper.markdown_rag_warmup`. Если warmup завершился timeout это нормальная ситуация при первом прогреве.

3. Для точных имён используй короткий query с идентификаторами: `limit=4`, `threshold=0.1`, `include_metadata=true`.

4. Для смысловых вопросов используй query по смыслу на русском или английском: `limit=5..7`, `threshold=0.1`, `include_metadata=true`.

5. RAG-результат используй как навигационный контекст. Если нужен точный ответ по реализации, уточняй его через исходный код после получения информации о структуре через RAG.

6. Если используешь `pasls`, сначала в этом же ответе вызови `pasls_diagnostics`. Не используй `pasls_references` и pasls-инструменты редактирования.

7. Если читаешь файл по `source_path` из RAG, считай `src/...` относительным путём внутри `D:/Works/учёба/вкр/finalversion/data/input_project`. Например `src/GpStringHash.pas` открывай как `D:/Works/учёба/вкр/finalversion/data/input_project/src/GpStringHash.pas`.

8. Если в wiki написано `Нет описания`, это не ошибка. Используй ID, объявление, видимость и `source_path`, затем при необходимости уточняй по исходнику.

9. Не выдумывай детали реализации. Если данных недостаточно, прямо скажи, что нужно проверить.

10. Не изменяй файлы, если пользователь просит только объяснение, анализ или поиск. Если просит изменить код — сначала кратко объясни план, затем вноси минимальные правки.

11. Отвечай на русском языке, кратко и по делу. В ответах по коду указывай:
- найденную сущность;
- unit/class/method;
- краткое назначение;
- где смотреть исходный код, если это полезно.
